# gatherEMR — 사용자 니즈 정본 (2026-07-04)

> 이 세션 전체에서 **사용자가 직접 입력한 프롬프트 전수**를 수집·정제한 요구사항.
> **전면 재작업(갈아엎기)의 기준 문서** — 구현·디자인 결정이 이 문서와 충돌하면 **이 문서가 이긴다.**
> 근거가 필요하면 맨 아래 [원문 프롬프트 로그](#부록-원문-프롬프트-로그)에서 확인.

## 0. 한 문장 정체성

**의대교수가 EMR 차트를 올리면, 자기 분과의 눈으로 요약해주는 사이트.**
요약은 믿으라고 주는 게 아니라 — **클릭 한 번으로 원본과 대조**하라고 주는 것.

## 1. 절대 원칙 (위반 = 실패)

1. **AI slop 제로 — 2축 검사.** 사용자가 보는 axis는 정확히 두 개:
   - **문구**: AI 말투스럽지 않은가? (개발자 눈에 "AI가 썼네" 바로 들킴 → 실패)
   - **디자인**: 획일화되지 않았는가? (Claude 전형 디자인 = 요즘 사이트 90%가 쓰는 그 디자인 → 실패)
   - 현재 상태 사용자 평가: **둘 다 0/100.** 이것이 갈아엎기의 직접 원인.
2. **로그인·가입 절대 없음.** 의대교수 귀차니즘 심함 — 가입 있으면 안 씀. 모두가 쓸 수 있어야 함.
3. **마찰 최소 = "예상 적중률 100%".** 교수들은 귀차니즘도 심하지만 **사용법을 잘 모른다.** 스크롤만으로 모든 맥락이 파악되고, "이걸 누르면 이게 되겠지"라는 직관적 예상이 **전부 맞게 작동**해야 한다. 접근성·마찰 감소의 정의가 이것. **레퍼런스 = 애플·토스가 하는 방식.**
3-1. **모든 작동은 인터랙티브 스텝 플로우.** 예: 드롭다운으로 자기 분과를 고르면 **화면이 오른쪽으로 이동**하며 다음 드롭다운/목록이 나온다 — 한 번에 한 선택, 진행감이 보이는 단계식 흐름 (토스식).
4. **업로드된 차트는 전부 내 Supabase에 저장** — 무기한.
5. **OpenAI 최상위 모델(gpt-5.5급) API** — 성능 최대한.
6. **완료 선언은 사용자 관점에서 실제 되는 것만.** (localhost "된다" 사건 재발 금지 — 내부 검증≠사용자 화면.)

## 2. 제품 요구

- **분과 선택 (v1 필수 — 응급의학과만 만드는 것 금지)**: **모든 분과** 의사·교수가 쓴다. **카테고리 셀렉트박스(드롭다운)**로 자기 분과를 고르면 **그 분과에 맞게 요약이 실제로 달라져야 함.** 26개 전부 UI에서 선택 가능 + 전부 작동. (2026-07-04 사용자 명시 재확인: "응급의학과만 만들어놨어" = 요구 위반.) 분과 목록은 대한민국 전문과목 전체(26개, 계열 카테고리화).
- **분과별 깊이 = 균등하게 전부 깊게.** "응급의학이 제일 깊고 나머지는 얕게" 방식 금지. **분과별로 의사 니즈가 다르다** — 각 분과가 어떤 부분 위주의 요약을 원하는지, 뭘 원할지를 **검색·자료조사·분석으로 파악한 후** 분과별 요약 설계에 반영한다. (2026-07-04 명시.)
- **스플릿뷰 + 하이퍼링크 (핵심 기능)**: 요약이 "왜 이렇게 요약됐는지" 보여주고, **요약 문장을 누르면 원본의 그 위치로 바로 점프.** 환각 문제는 이 구조로 — 인간이 원본을 직접·즉시 판단.
- **입력은 뭐든**: PDF·사진·텍스트 전부 가능해야 함 → OCR 필요. **OCR 성능 중요.** 한글=업스테이지가 가장 정확(인정), 단 영어 위주 문서 대비 필요.
- **입력 UX (귀차니즘 계층 설계)**: ① 아무것도 안 해도 그냥 바로 진행 가능 ② 버튼식 예시 2~3개("이런 거 위주로 요약해주세요" 류 — 의사 니즈 기반으로 도메인 지식·검색으로 정할 것) ③ 원하면 한 줄 직접 입력(플레이스홀더).
- **동의 문구**: 차트요약 누를 때 간략한 동의 문구 한 개.
- **v1 목표 = 정식 제품** (데모·포트폴리오 아님). 멀티유저.
- **품질 데이터**: gold 케이스 = 공개 케이스. slop-린트 = 규칙 기반.

## 3. 랜딩 요구

- **사용설명서 금지.** 스크롤 슥슥 내리면 **바로 파악**되게 — 짧고 간결하게.
- **샘플데이터로 보여주기**: 일반적인 타입 몇 개만 정해서(PDF/사진/텍스트 중) "이걸 넣으면 → 이런 식으로 요약" 즉시 체험.
- 기능 설명의 핵심 = **스플릿뷰+하이퍼링크 이동** (환각을 인간이 원본 판단으로 잡는 구조).
- **마무리 섹션 = "고도화 진행 중"**: 분과별 요약·의학 약어 통합을 깊게 파고 있다는 신호로 끝낼 것.
- **정보량 최소**: "정보가 너무 많아. 전부 다 안 볼 거임." — 지금 랜딩의 실패 원인 중 하나.

## 4. 디자인 요구

- **⚠️ 시퀀스 (2026-07-04 확정): 지금은 단순 MVP — 디자인에 힘쏟지 않는다.** 먼저 **요약이 잘 작동**해야 한다. 디자인은 **모든 페이지가 작동하는 걸 본 후** 마무리 작업으로. (단, §1-3의 인터랙티브 스텝 플로우·직관 작동은 디자인이 아니라 **작동 방식**이므로 MVP에 포함.)
- **시네마틱** (finale 단계): 영상이나 이미지를 **다운로드하거나 직접 만들어서** 시네마틱하게. (CSS 텍스트 장식만으로는 부족했음.)
- **심미적으로 이쁘고** 접근성 좋게 — 단 1번 원칙(2축 anti-slop)이 항상 우선.
- 획일화 판정 기준: "요즘 만들어지는 사이트 90%가 쓰는 그 디자인"이면 탈락. Claude가 뽑는 전형이면 탈락.
- 문구 판정 기준: 개발자가 봤을 때 "AI 느낌" 나면 탈락.
- **디자인은 마지막 중요 작업으로 분리** — 기능 전부 완료 후 finale로 진행하되, 이번엔 이 문서 기준으로.

## 5. 협의로 확정된 결정 (사용자 승인 이력)

- **경로 A: 비식별 전용 + 능동 스크럽** — "주로 안전한 데이터" 전제를 설계로 강제. (법적 부분은 카운슬 과하게 하지 말 것 — 기획·퀄리티에 포커스하라고 명시 지시.)
- 보관정책 = **무기한 저장**.
- 요약 언어 = 한글 서술 + 의학용어는 임상 표기 그대로(SOB, Troponin, r/o ACS).
- v1 검증-딥 분과 = 응급의학과 1개, 나머지 25개 = 선택 가능 + "정밀화 진행 중".
- 이미지·스캔 OCR = Phase 2 (v1 = 텍스트/텍스트PDF).
- 디자인 v2(시네마틱) 방향 채택 — 단 위 4번 기준으로 재작업 대상.
